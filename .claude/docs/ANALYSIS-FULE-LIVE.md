# 富樂 LIVE 深度分析（競品研究）

> 分析日期：2026-07-02　來源：https://lab.lamigifts.com/fule23/（公開原始碼靜態分析＋會員視角實測）
> 目的：作為富禮分會系統設計的參考——學它的優點，避開它的坑

---

## 一、定位

「富樂 LIVE」是富樂分會**第 23 屆小組競賽平台**：把分會 KPI（出席、引薦、一對一、培訓、繳費）包裝成 11 個小組的積分競賽，冠軍獎金 $6,000。本質是**遊戲化的營運儀表板**，不處理任何「行為本身」（不能約一對一、不能填引薦單），只做記錄、統計與激勵。

## 二、技術架構

| 層 | 實作 | 備註 |
|----|------|------|
| 前端 | 單一 HTML 檔（498KB），React 18 UMD via CDN，無建置步驟，全部 `React.createElement` | 30 個元件全部 inline |
| 動態資料 | Firebase Realtime Database（`scores`、`board` 留言板、`taunts` 戰報、`announces`、`calendar`、`bonuses`、`trainingZone`、`groupNames`、`monthMbrCount`） | 前端直連 |
| 靜態資料 | **硬編碼在 HTML**：成員名單（GROUPS/ALL_MBR，85 人＋專業別）、PALMS 統計（PALMS_RAW，約 5KB）、紅綠燈歷月資料（TLD）、入會週次（MEMBER_SINCE） | 每次 PALMS 更新＝改檔重佈署 |
| 表單 | Google Apps Script（FORM_API）＋獨立表單頁 | 資料填寫模組 |
| 檔案 | Firebase Storage（留言板圖片）、SheetJS 讀 Excel（PALMS 匯入工具） | |
| 權限 | 前端密碼比對：`9999`=demo（唯讀展示）、SPW=super、CPW=核心、LPW=各組組長各一組密碼 | **所有密碼硬編碼在公開 HTML 內** |

### 維運模式（關鍵洞察）

整套系統依賴一位維護者（署名「富樂花花 × AI 協作」）：每週用 AI 協作改 HTML（更新 PALMS_RAW、紅綠燈、名單）→ 重新上傳。組長每週登入填各組積分，核心團隊管理公告/行事曆/加分。**這是「一個熱血會員＋AI」的產物，不是工程團隊的產物**——它能動，但複製或交接的成本很高。

### 安全問題（我們必須避開的坑）

1. 所有角色密碼（含各組組長密碼）都在公開原始碼裡，任何人看 source 就能拿到寫入權
2. Firebase 由前端直連，資料庫規則形同全開——85 位成員的姓名、專業別、出席率、引薦數等 PALMS 個資**公開可讀**
3. demo 密碼（9999）外流到外分會流通，也印證權限只是君子協定

## 三、功能清單與熱度評估

| 功能 | 內容 | 價值評估 |
|------|------|---------|
| 小組排行榜 | 11 組積分＋展開成員明細＋積分明細帳 | ★★★ 競賽核心，做得好 |
| LIVE 戰報 (WarTicker/TauntBoard) | 「XX 帶領第 8 組超越第 5 組！」跑馬燈＋組間嗆聲板 | ★★★ 黏著度引擎，巧思 |
| 會員光榮榜 | 個人積分前 30 頒獎台（星空頒獎 UI） | ★★☆ 榮譽感 |
| 個人戰績 | 按組查每人每週明細 | ★★☆ |
| 怎麼得分 | 計分規則說明（咖啡會議 20/場、帶來賓 15、簽單 15、宣誓 15、早鳥 7；組聚 15、早安貼圖 15、全員綠燈 50、繳費 10；咖啡會議多人有乘數表） | ★★☆ 必要配套 |
| 紅綠燈表 | 歷月各組成員燈號（BNI 中心區制度的鏡像） | ★★★ 分會真實剛需 |
| 預測綠燈 | 用 PALMS 快照試算「再衝 12 位達 80%」＋列出補救名單（差一對一/引薦就轉綠的人） | ★★★ **全站最有價值的功能**——把數據變成行動指令 |
| 培訓專區 | 培訓率追蹤（目標 12%） | ★★☆ |
| 匿名留言板 | 匿名貼文＋圖片＋愛心＋回覆 | ★★☆ 意外活躍，社群黏著 |
| 資料填寫追蹤 | Google Form 完成度＋最速小組加分 | ★☆☆ 屆期性 |
| 產業服務鏈＋25秒順序 | 按產業鏈分類的會員名錄 | ★★☆ 名錄剛需 |
| 重要公告/行事曆 | 靜態列表 | ★☆☆ LINE 群可取代 |
| 我的事業在BNI的目標 | 目標計算機：年營業額→反推每週一對一次數（BNI 大數據 1.5 張引薦/次） | ★★☆ 一次性但有說服力 |

### 值得學的三件事

1. **「預測＋補救名單」思維**：不只給數字，直接告訴你「這 20 人補一場一對一就綠燈」——數據變行動
2. **戰報跑馬燈**：把枯燥的積分變成有情緒的敘事（超越、暴增 414%）
3. **零登入的角色分級**：會員看不用登入，門檻極低（雖然實作方式不安全，但「低門檻」方向是對的）

### 它的結構性弱點（我們的機會）

1. **只記錄、不促成**：它算出「你每週要 3 次一對一」，但你還是得回 LINE 慢慢約——行為斷點正是我們預約平台要接的
2. **維運重**：靠一個人每週手改檔案；換屆或那個人離開，系統就死
3. **個資裸奔**：如上
4. **屆期綁定**：整包跟「第 23 屆競賽」綁死（fule23），賽季結束即棄置，資產不累積

## 四、對富禮版的設計含義

- 我們的核心差異化＝**促成行為**（一對一預約引擎），這是富樂完全沒有的
- 富樂證明了：排行榜＋戰報＋留言板的遊戲化組合在 BNI 文化裡確實有效
- 瘦身原則：砍掉 LINE 群可取代的（公告/行事曆）、屆期性的（資料填寫追蹤）；保留數據剛需（紅綠燈/預測綠燈）與黏著引擎（排行/戰報），且全部改成**可持續維運**的架構（資料庫＋權限，不是手改 HTML）
